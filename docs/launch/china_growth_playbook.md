# Fdesign 国内增长作战手册

更新日期：2026-06-22

长期目标：把 Fdesign 运营成 10,000 Star 级别的真实开源项目。第一阶段不是追求“一波爆”，而是先拿到 100 个真实使用者、10 个真实问题、1 个可公开示例包，再逐步放大到 1,000 Star 和 10,000 Star。

- 远端状态：public
- 2026-06-22 复查：`0 stars / 2 forks`

## 国内推广原则

- 不买 Star、不互刷、不诱导点赞收藏，不把 Star 作为交换条件。
- 国内内容主张“解决一个真实作图流程”，不喊“神器”“免费领取”“限时福利”。
- GitHub 是 Star 主阵地；国内平台负责教育用户、解释场景、收集问题和引导试跑。
- 每个平台只在有新内容、新案例、新教程或新版本时发布，避免重复刷帖。
- 不发布私有 PSD、付费字体、真实商品数据、内部业务资料或未净化截图。
- 小红书、视频号、抖音等平台不做硬导流话术，不放二维码式外链轰炸；优先用“项目名 + 开源 + GitHub”让用户主动搜索。

## 渠道打法

### V2EX

定位：面向独立开发者和工具用户的“开源项目展示 + 求反馈”。

打法：

- 首帖放在分享创造/程序员/设计相关节点中最匹配的节点。
- 标题必须说明具体工作流，不用标题党。
- 正文先讲限制：Windows + Photoshop + Node.js、本地运行、不含模板素材。
- 只在版本更新、示例包、技术复盘时再发后续帖。

验收：

- 有真实回复、issue、clone 或 referrer。
- 没有因重复推广被反感或删帖。

### 掘金

定位：技术搜索入口，承接 React + Node.js + Photoshop 自动化开发者。

打法：

- 文章标题用“开源 + 技术链路 + 具体结果”。
- 第一篇写整体架构，后续拆分 PSD 解析、Photoshop 调度、导出稳定性、开源净化。
- 每篇只放一次项目页和 GitHub 链接，避免像广告。

建议系列：

1. 我把电商 PSD 批量作图工作台开源了：React + Node.js 调 Photoshop 的本地自动化实践
2. Node.js 如何稳定调起 Photoshop 导出 PSD/PNG/JPEG
3. PSD 模板变量绑定怎么做：从 Excel 字段到 Photoshop 图层
4. 开源一个本地工具前，我如何清理私有素材和运行产物
5. Fdesign V3.0 的拆包优化：从 1.7MB 主包到稳定 vendor chunks

### 知乎

定位：长尾问答搜索。

打法：

- 不写“项目安利”式回答，围绕具体问题回答。
- 回答结构：问题背景 -> 普通做法缺陷 -> 本地自动化思路 -> Fdesign 示例 -> 限制条件。
- 适合问题方向：
  - 电商详情页图片如何批量生成？
  - Photoshop 能否批量替换 PSD 里的智能对象？
  - Excel 数据怎么批量套进设计模板？
  - 小团队如何搭建本地图片生产自动化？

### B 站

定位：真实工作流演示，降低理解成本。

打法：

- 第一条视频 60-90 秒，必须展示“Excel -> PSD 预览 -> 绑定 -> 导出结果”。
- 标题强调结果，不写过度承诺。
- 简介放项目页和 GitHub；评论区只补充运行条件和 FAQ。
- 发布后必须用平台侧页面或创作中心验证公开状态。

建议选题：

- 把 Excel 商品数据批量套进 PSD，我把这个 Photoshop 工作台开源了
- 设计师最怕的重复改图，我用本地 Photoshop 自动化解决了一半
- 一个开源工具如何批量导出 PSD、PSB、PNG、JPEG

### 小红书

定位：设计师和电商运营的痛点卡片，不强导流。

打法：

- 用 6-8 张图讲“重复改图痛点 -> 本地工作台 -> 输出结果 -> 限制条件”。
- 不放二维码，不写“评论区领取”，不做诱导互动。
- 结尾只写“项目名 Fdesign，已开源，可搜索 GitHub 仓库”，把链接放到平台允许的位置。

### 视频号 / 微信社群 / 公众号

定位：真实业务人群触达和反馈收集。

打法：

- 社群短帖先发给电商设计、运营、独立开发者小圈层。
- 公众号适合写“开源复盘 + 工作流说明”，不要做销售页。
- 视频号只做低风险工作流演示，不展示敏感桌面、账号、后台、订单或支付画面。

### 开源中国 / Gitee / GitCode

定位：降低国内开发者访问和发现门槛。

打法：

- 建议准备只读镜像或同步说明，但 Star 目标仍以 GitHub 为准。
- 镜像 README 顶部说明：主仓库和 Issues 在 GitHub。
- 不把镜像当分叉主阵地，避免反馈分散。

## 国内首月执行顺序

1. 补齐一套可公开演示包：示例 PSD、示例 Excel、示例商品图、导出结果截图。
2. 发布 V2EX 首帖，重点求 README/安装/模板兼容反馈。
3. 发掘金第一篇技术文章，把技术可信度打出来。
4. 录 60-90 秒 B 站演示，链接项目页和 GitHub。
5. 发小红书图文，把“重复改图痛点”讲给设计师。
6. 把所有反馈收敛到 GitHub Issues/Discussions。
7. 每周运行 `scripts/capture_github_growth_metrics.ps1`，记录 Star、views、clones、referrers。

## 停止规则

- 某平台出现删帖、限流或明显反感，暂停该平台重复发布，只保留复盘和后续实质更新。
- 如果某渠道只带来浏览、不带来 clone/issue，下一轮内容改成安装教程或真实案例。
- 如果出现“能不能给模板/素材/成品服务”的请求，统一引导到开源边界说明，不让服务请求干扰开源主叙事。

# Fdesign V3.0 First 30 Days Growth Plan

更新日期：2026-06-16

目标：用真实使用者、真实反馈和真实传播，把 Fdesign 从公开发布基线推向 10,000 stars。当前长期目标没有完成；第一阶段里程碑是 1,000 stars，只有 GitHub 显示 10,000+ stars 且复查无异常时才算达成长期目标。

## North Star

- GitHub stars：反映收藏和后续试用意愿。
- Unique views：判断分发是否带来真实访问。
- Unique clones：判断是否有人开始本地试跑。
- Issues / Discussions：判断是否有真实问题和模板场景。
- README -> Star 转化：`新增 stars / unique views`，按发布窗口粗略估算。

## Week 1：拿到第一批真实问题

| 时间 | 动作 | 目标 | 验收 |
| --- | --- | --- | --- |
| Day 0 | 确认 GitHub Pages、README、Release、Issues、Discussions 可访问 | 所有公开入口可用 | `scripts/setup_github_growth.ps1` 可重复运行 |
| Day 0 | 国内电商设计师/运营社群短帖 | 找到最贴近痛点的人 | 至少 5 条反馈或 2 个 issue |
| Day 1 | V2EX 首发 | 获得开发者/独立开发者反馈 | 记录 referrer、views、stars |
| Day 2 | 补 README FAQ | 降低首次运行阻力 | 新增常见问题不超过 8 条，避免冗长 |
| Day 3 | 掘金技术拆解 | 沉淀搜索流量 | 文章评论里收集技术问题 |
| Day 4-7 | 修首批文档/安装问题 | 把反馈变成可见改进 | 至少 1 个文档 PR 或提交 |

| 动作 | 目标 | 验收 |

| 月度复盘 | 找到下一轮增长瓶颈 | 输出 stars、views、clones、referrers、issue 质量 |

## 每周复盘问题

- 哪个渠道带来最多 unique views？
- 哪个渠道带来最多 clones 或 issue？
- README 哪一段被反复问到？
- 是否有用户愿意提供净化后的模板场景？
- Star 增长来自真实访问，还是只是熟人扩散？
- 下周最该补的是文档、案例、视频，还是兼容性？

## Stop Rules

- 如果某渠道出现明显反感或删帖，停止重复发布，只保留后续技术复盘。
- 如果 Star 增长但 clones/issues 为 0，优先补使用门槛和案例，不继续扩大投放。
- 如果发现截图或文案含私有信息，立即下架对应素材并替换。
